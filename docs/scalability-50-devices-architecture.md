# 50대+ 디바이스 확장 아키텍처 제안

## 개요

**작성일**: 2026년 2월 3일
**상태**: 제안 (미구현)
**목표**: 현재 10대 동시 실행 한계를 50대 이상으로 확장하기 위한 아키텍처 설계

---

## 현재 시스템 한계

### 병목 지점 분석

| 병목 | 현재 상태 | 영향 |
|------|----------|------|
| **OpenCV 이미지 매칭** | 단일 프로세스, 세마포어 4개 제한 | CPU 과부하 |
| **Appium 서버** | 단일 서버 (포트 4900) | 세션 관리 한계 |
| **백엔드 서버** | 단일 Node.js 프로세스 | 요청 처리 한계 |
| **작업 분배** | 직접 호출 (큐 없음) | 부하 분산 불가 |

### 현재 지원 규모

| 항목 | 수치 |
|------|------|
| 디바이스 목록/UI | 20대+ |
| **안정적 동시 테스트** | **10대** |
| 이미지 매칭 동시 처리 | 4개 |

---

## 권장 아키텍처 (50대 기준)

```
                         ┌─────────────────┐
                         │  Load Balancer  │
                         │  (Nginx)        │
                         └────────┬────────┘
                                  │
              ┌───────────────────┼───────────────────┐
              │                   │                   │
       ┌──────▼──────┐     ┌──────▼──────┐     ┌──────▼──────┐
       │ Backend #1  │     │ Backend #2  │     │ Backend #3  │
       │   (PM2)     │     │   (PM2)     │     │   (PM2)     │
       └──────┬──────┘     └──────┬──────┘     └──────┬──────┘
              │                   │                   │
              └───────────────────┼───────────────────┘
                                  │
                         ┌────────▼────────┐
                         │   Redis Queue   │
                         │   (BullMQ)      │
                         └────────┬────────┘
                                  │
         ┌────────────────────────┼────────────────────────┐
         │                        │                        │
  ┌──────▼──────┐          ┌──────▼──────┐          ┌──────▼──────┐
  │ ImageMatch  │          │ ImageMatch  │          │ ImageMatch  │
  │  Worker 1   │          │  Worker 2   │          │  Worker 3   │
  └─────────────┘          └─────────────┘          └─────────────┘

         ┌────────────────────────┼────────────────────────┐
         │                        │                        │
  ┌──────▼──────┐          ┌──────▼──────┐          ┌──────▼──────┐
  │  Appium #1  │          │  Appium #2  │          │  Appium #3  │
  │  (17대)     │          │  (17대)     │          │  (16대)     │
  └─────────────┘          └─────────────┘          └─────────────┘
```

### 핵심 구성 요소

| 구성 요소 | 역할 | 기술 스택 |
|----------|------|----------|
| Load Balancer | 요청 분산 | Nginx / HAProxy |
| Backend Cluster | API 처리 | Node.js + PM2 |
| Redis Queue | 작업 큐 관리 | Redis + BullMQ |
| Image Workers | OpenCV 매칭 | Worker Threads / 별도 서비스 |
| Appium Cluster | 디바이스 제어 | Appium × N |

## 단계별 구현 로드맵

### Phase 1: 즉시 적용 (10대 → 20대)

| 작업 | 설명 | 소요 시간 |
|------|------|----------|
| ✅ 세마포어 + 템플릿 캐싱 | 완료 | - |
| PM2 클러스터 모드 | `pm2 start app.js -i max` | 1일 |

### Phase 2: 중기 개선 (20대 → 35대)

| 작업 | 설명 | 소요 시간 |
|------|------|----------|
| Worker Threads | OpenCV를 별도 스레드로 분리 | 3-5일 |
| Appium 분산 | 디바이스 그룹별 Appium 서버 | 2-3일 |
| 라우팅 로직 | 디바이스 → Appium 매핑 | 1-2일 |

**예상 소요: 1-2주**

### Phase 3: 본격 확장 (35대 → 50대)

| 작업 | 설명 | 소요 시간 |
|------|------|----------|
| Redis 도입 | 작업 큐 중앙화 | 2-3일 |
| BullMQ 적용 | 작업 분배 및 재시도 | 3-5일 |
| 별도 매칭 서비스 | Python/FastAPI 또는 Node.js 분리 | 1주 |
| Appium 확장 | 3-5개 서버로 확장 | 2-3일 |

**예상 소요: 3-4주**

### Phase 4: 대규모 (50대 → 100대+)

| 작업 | 설명 | 소요 시간 |
|------|------|----------|
| Kubernetes | 컨테이너 오케스트레이션 | 2-4주 |
| Auto-scaling | 부하 기반 자동 확장 | 1-2주 |
| 모니터링 | Prometheus + Grafana | 1주 |

**예상 소요: 2-3개월**

## 머신 성능 요구사항

### 구성 요소별 리소스

| 구성 요소 | CPU | RAM | 비고 |
|----------|-----|-----|------|
| Backend (PM2 × 3) | 낮음 | 600-900MB | I/O 바운드 |
| Redis | 매우 낮음 | 500MB-1GB | 작업 큐 + 캐시 |
| Image Workers (× 3) | **매우 높음** | 1.5-3GB | CPU 집약적 |
| Appium (× 3) | 중간 | 900MB-1.5GB | 세션 관리 |
| 스크린샷 버퍼 | - | 1-1.5GB | 50대 × 20MB |

### 규모별 권장 사양

#### 온프레미스

| 규모 | CPU | RAM | 스토리지 | 예상 비용 |
|------|-----|-----|----------|----------|
| 20대 | 8코어 | 16GB | SSD 256GB | 기존 PC 활용 |
| 35대 | 8코어 | 32GB | SSD 512GB | 100-150만원 |
| **50대** | **12코어+** | **32-64GB** | **SSD 1TB** | **200-300만원** |

**권장 CPU:**
- Intel: i7-12700 / i9-12900 / Xeon E-2300
- AMD: Ryzen 7 5800X / Ryzen 9 5900X

#### 클라우드 (AWS 서울 리전)

| 규모 | 인스턴스 | 사양 | 월 비용 |
|------|----------|------|--------|
| 20대 | t3.xlarge | 4코어 / 16GB | ~$150 |
| 35대 | m5.2xlarge | 8코어 / 32GB | ~$300 |
| 50대 | c5.4xlarge | 16코어 / 32GB | ~$500 |
| 50대 분산 | t3.xl × 2 + c5.2xl | API + Worker | ~$450 |

## 추가 인프라 고려사항

### 디바이스 연결 방식

| 방식 | 장점 | 단점 | 권장 규모 |
|------|------|------|----------|
| **USB 허브** | 안정적, 지연 없음 | 물리적 한계, 전원 이슈 | ~20대 |
| **ADB over TCP** | 무제한 확장 | 네트워크 부하, 약간의 지연 | **20대+** |

**50대 규모에서는 ADB over TCP 권장:**
```bash
# 디바이스에서 TCP 모드 활성화
adb tcpip 5555
adb connect 192.168.1.x:5555
```

### 네트워크 요구사항

| 항목 | 요구사항 |
|------|---------|
| 대역폭 | 1Gbps |
| 지연 | <10ms (같은 LAN) |
| 연결 | 유선 권장 |
